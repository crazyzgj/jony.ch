Configuring V-STP

Context

Virtual Spanning Tree Protocol (V-STP) is a Layer 2 topology management feature and virtualizes two STP-enabled devices into one device to perform STP calculation.

STP can detect the M-LAG master or backup status. After V-STP is enabled on the M-LAG master and backup devices and M-LAG master/backup negotiation is successful, two devices are virtualized into one device for port role calculation and fast convergence. STP needs to synchronize the bridge MAC addresses and instance priorities of the M-LAG master and backup devices. After M-LAG master/backup negotiation is successful, the backup device uses the bridge MAC address and instance priority that is synchronized from the master device for STP calculation and packet transmission. This ensures STP parameter calculation on the virtualized device.

V-STP can be only applicable to M-LAG networking. It can be used in multi-level M-LAG interconnection scenarios and scenarios where devices in the M-LAG function as non-root-bridges.

When configuring V-STP, ensure that the STP/RSTP timer settings on the two devices that constitute an M-LAG be the same. Otherwise, network flapping may occur.

Procedure

  1. Run system-view

    The system view is displayed.

  2. Run stp mode { stp | rstp }

    The switch is configured to work in STP or RSTP mode.

    By default, the switch works in MSTP mode.

    V-STP does not support the MSTP mode, and supports multi-process. By default, an MSTP process works in MSTP mode. Currently, only STP and RSTP modes are supported in V-STP scenarios. The MSTP process therefore must be configured to work in STP or RSTP mode in V-STP scenarios.

  3. (Optional) Run stp bridge-address mac-address

    The bridge MAC address used in spanning tree participation is configured.

    By default, the switch's MAC address is the bridge MAC address of the switch that participates in spanning tree calculation.

    In a V-STP scenario, an M-LAG master/backup switchover may cause slow STP convergence. You are advised to configure a bridge MAC address when the instance priorities of the two M-LAG devices are the same. The two M-LAG devices must be configured with the same bridge MAC address. You can select the system MAC address (display system mac-address) of one device as the bridge MAC address of the two devices.

  4. (Optional) Run stp auto-cost-incr autoCostIncr

    The path cost increment is configured.

    By default, no path cost increment is configured.

    • After this command is configured, the path cost of a port is the original cost plus the increment configured using this command.
    • After this command is configured, the path costs that are calculated using a standard method on all STP ports are adjusted and all spanning trees are recalculated, which may cause service flapping.
    • This command takes effect globally after being configured. If the stp cost command is configured on a port, the cost configured on the port takes effect.
    • In an M-LAG configured in V-STP mode, if an Eth-Trunk interface is configured as an M-LAG member interface, the path cost of the interface is fixed at 2000. When configuring the stp auto-cost-incr command, you are advised to set the parameter to a value greater than or equal to 2000.

  5. Run stp v-stp enable

    V-STP is enabled on an M-LAG device.

    By default, V-STP is disabled on an M-LAG device.

  6. (Optional) Run stp v-stp port-id-extension enable

    The M-LAG member interface is configured to send BPDUs with an extended port ID.

    By default, the port ID carried in BPDUs sent by the M-LAG member interface in V-STP mode is 1.

    The default port ID carried in BPDUs sent by the M-LAG member interface is 1. After the downstream device interface is configured to transparently transmit packets, BPDUs sent by the M-LAG member interface are looped back to the two V-STP-enabled devices. In this case, the system MAC address and port ID in BPDUs received by the M-LAG member interface are the same as those of the interface. As a result, the M-LAG member interface is incorrectly blocked. To solve this problem, you can configure the M-LAG member interface to send BPDUs with an extended port ID that is not the default port ID 1. In this case, the port ID in BPDUs received by the M-LAG member interface is different from that of the interface, and the M-LAG member interface is not blocked.

  7. Run commit

    The configuration is committed.
