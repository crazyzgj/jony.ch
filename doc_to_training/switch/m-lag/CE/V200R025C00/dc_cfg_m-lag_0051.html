M-LAG Upgrade in Maintenance Mode Fundamentals

By running a command for each routing protocol used on a device (OSPF/OSPFv3/BGP/BGP4+) and another command for LACP in the maintenance mode view before upgrading the device, you can switch traffic to the secondary link. After the upgrade is complete, you can switch traffic back by deleting the previous configurations (delete the LACP configuration first and then each routing protocol configuration). Then, exit the maintenance mode view. Compared with conventional upgrade, M-LAG upgrade in maintenance mode improves network reliability.

Typical application scenarios of M-LAG upgrade in maintenance mode include M-LAG dual-homing access node upgrade, spine node M-LAG upgrade, VXLAN M-LAG dual-homing access node upgrade, and multi-level M-LAG upgrade.

M-LAG Dual-Homing Access Node Upgrade

In Figure 1, SwitchG to SwitchJ are each dual-homed to an M-LAG. Their uplink interfaces are ECMP-enabled Layer 3 interfaces with one or more routing protocols (OSPF/OSPFv3/BGP/BGP4+) configured.

To perform an M-LAG upgrade on SwitchA, you must first switch traffic to SwitchB. Conventionally, this would be done by shutting down service interfaces. After the switchover is complete, you can then restart and upgrade SwitchA. Following this, you need to switch traffic back to SwitchA by restoring the service interface states. However, during the switchover, SwitchE detects uplink interface state changes and updates ECMP paths, resulting in the loss of north-to-south traffic. Similarly, SwitchG detects state changes of M-LAG member interfaces and updates LACP interface states, resulting in the loss of south-to-north traffic. In addition, due to a time difference between the state changes of Eth-Trunk 1 and Eth-Trunk 2, east-west traffic will also be lost when LACP interface states are updated.

Figure 1 M-LAG dual-homing access node upgrade (conventional mode)

Figure 2 shows M-LAG upgrade in maintenance mode, in which SwitchA enters the maintenance mode view before it is restarted for an upgrade.

  • Configure SwitchA to change the cost values of OSPF/OSPFv3 routes or the MED and Local_Pref values of BGP/BGP4+ routes so that traffic sent from the peer device to SwitchA can be switched to SwitchB.
  • Enable SwitchA to forcibly set Eth-Trunk interfaces that are added to an M-LAG to down. SwitchA then sends dying gasp packets to SwitchG and SwitchH. Upon receipt of these packets, SwitchG and SwitchH switch traffic to other interfaces.

After performing the preceding operations in the maintenance mode view, you can upgrade the software version of SwitchA.

Once the upgrade is complete, you need to switch service traffic back to SwitchA and exit the maintenance mode view to implement M-LAG upgrade in maintenance mode.

  • Disable SwitchA from forcibly setting Eth-Trunk interfaces that are added to an M-LAG to down. SwitchA then sends negotiation packets to SwitchG and SwitchH. After receiving the packets, SwitchG and SwitchH switch traffic back.
  • Configure SwitchA to restore the cost of OSPF/OSPFv3 routes or the MED and Local_Pref of BGP/BGP4+ routes to the default values. This enables traffic sent from the peer device to SwitchA to be switched back.
Figure 2 M-LAG dual-homing access node upgrade (maintenance mode)

Multi-level M-LAG upgrade

In Figure 7, the server is dual-homed to an M-LAG. An M-LAG is configured between SwitchA and SwitchB, and between SwitchC and SwitchD. One or more routing protocols (OSPF/OSPFv3/BGP/BGP4+) run between SwitchC and SwitchD and network-side devices.

To perform an M-LAG upgrade on SwitchA, you must first switch traffic to SwitchB. Conventionally, this would be done by shutting down service interfaces. After the switchover is complete, you can then restart and upgrade SwitchA. Following this, you need to switch traffic back to SwitchA by restoring the service interface states. However, during the switchover, the peer device detects state changes of M-LAG member interfaces and updates the LACP interface states, resulting in packet loss.

Figure 7 Multi-level M-LAG upgrade (conventional mode)

Figure 8 shows M-LAG upgrade in maintenance mode, in which SwitchA enters the maintenance mode view before it is restarted for an upgrade.

After performing the preceding operations in the maintenance mode view, you can upgrade the software version of SwitchA.

Once the upgrade is complete, you need to switch service traffic back to SwitchA and exit the maintenance mode view to implement M-LAG upgrade in maintenance mode.

  • Configure SwitchA to restore the cost of OSPF/OSPFv3 routes or the MED and Local_Pref of BGP/BGP4+ routes to the default values. This enables traffic sent from the peer device to SwitchA to be switched back.
